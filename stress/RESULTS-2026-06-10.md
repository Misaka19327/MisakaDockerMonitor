# Stress Test Results (2026-06-10)

This note captures the current stress-test baseline after the log pipeline pressure reduction work and the raw-first parsing change.

## Test setup

- Monitor image: current local workspace build as of 2026-06-10.
- Storage backends tested: `sqlite`, `mysql`.
- Load generator: `stress/loadgen`.
- Sampling source: `/api/debug/runtime` plus `/api/containers`.
- Host environment:
  - Dockerized monitor instances were used for reliable Docker socket access.
  - Direct Windows-host Bun runs were not used for final numbers because `dockerode` against `//./pipe/docker_engine` was unstable in this environment.

## Results summary

### SQLite, 50 containers x 100 logs/s

- Approximate ingest rate: `~5000 logs/s`
- Sampling duration: `60s`
- API latency:
  - average: `889.52ms`
  - p95: `1307.62ms`
  - max: `1307.62ms`
- Process memory:
  - RSS max: `239.77MB`
  - heap used max: `135.31MB`
- Collector pressure:
  - buffered entries max: `455`
  - queued write entries max: `7579`
  - flush duration max: `225ms`
- Integrity:
  - dropped entries: `0`
  - flush errors: `0`
- Interpretation:
  - still clearly degraded, but basically usable
  - current best backend for this code path

### SQLite, 100 containers x 100 logs/s

- Approximate ingest rate target: `~10000 logs/s`
- Full scripted sampling did not complete cleanly because API requests timed out for more than `120s`.
- `docker stats` during the run showed approximately:
  - CPU: `201.06%`
  - memory: `3.384GiB`
- Interpretation:
  - process stayed alive
  - API was effectively no longer usable
  - current version should not be considered safe for sustained `100 x 100 logs/s`

## Current capacity estimate

- Conservative "basically usable" upper bound:
  - `~4000-5000 logs/s`
  - for example `40-50 x 100 logs/s`
- Estimated edge-of-usable upper bound:
  - `~6000-7000 logs/s`
  - based on interpolation between the `5000 logs/s` and `10000 logs/s` runs
- Not currently acceptable:
  - `~10000 logs/s`

## Architectural takeaway

- Moving synchronous parsing out of the backend hot path helped.
- The next dominant bottleneck is write throughput and queue growth, not log parsing itself.
- In the current code, SQLite is the better backend for this small internal tool.
- If higher sustained throughput is required, the next likely improvements are:
  - multi-writer or partitioned write workers
  - explicit queue degradation rules when backlog crosses a threshold
  - optional asynchronous parse backfill if full historical level/group semantics must be preserved
